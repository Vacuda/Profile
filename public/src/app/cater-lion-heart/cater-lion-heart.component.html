General

 

Game Tag Line

▼

    The Talismans Of Genyatium is a calm, flow-inducing, procedurally-generated, multiple-puzzle, jigsaw puzzle game based on hexagons and triangles and ... grief. You find the pieces in containers piled up in a typical level, no enemies, no deaths, just lots of things to break and lots of things to put back into place.

Procedural Generation Design

▼

    From the beginning, this game idea needed to be procedurally-generated in a number of different ways. I knew that this would complicate the development process, but I just trusted that I would be able to figure it out when I got there. Knowing this from the beginning helped me to frame every decision. I could build something as a placeholder, but if I knew it would need to be dynamic later, I consistently had to think about building the foundation that would enable that.

 

    What you learn about procedural generation is not really to build something that has no constraints, but to build something with specific and consistent constraints, in which you can fill the spaces in with randomness.

 

    I got it working great! There's much work still to be done to make the physical levels more appealing, but as for the dynamic puzzles, the foundation is strong. Some variables already effectively alter how a world is built, and the plan is to just add more, making puzzle worlds entirely customizable.

Learning Unreal Engine

▼

    I am developing this game as I learn the Unreal Engine. I wanted to jump into the industry standard engine to expose myself to the kind of tool I'd be working with on the job. While I've come a long way, this thing is massive and obtuse for a beginner. The more you know, the more you know you don't know. And while I'm familiar with many pieces of software, its been a journey navigating Unreal and learning it as a language itself, let alone connecting it to my C++ code.

Code Design

 

C++ Integration

▼

    It was important to me to learn to write a game in C++ code rather than blueprints. In fact, busting out a Material Blueprint was more intimidating than working with C++ code. But it took awhile to really learn(and I'm still learning, with every new UE4 tool I use) how to connect the C++ code to the editor.

    In a lot of ways, this has been my biggest obstacle in learning UE4. You have to learn how to exchange references between code and editor. You have to understand the object inheritance trees and how and when to use which. You have to understand how and when to instantiate actors or objects and why.

Communication Between Everything

▼

    I'm big into code organization, so I was creating new classes to house different sections of code. Quickly, this meant that any time I needed one class to talk to another, I needed to first hook them up to each other. This was becoming cumbersome each time, so I created the Hub. This was a housing class that would be built at the beginning of a level and it housed all of the relevant connections. Now, a class only needs to hook up to the Hub, and it can start to talk to eveyone.

    I'm currently rethinking this, though. The Hub builds and houses all of the classes I created to organize code, but as I move away from OOP, I will be translating some of those classes to namespaces and static functions. Now, instead of Hub-> BoardArbiter-> RotateThisPiece(GrabbedPiece), I'll be able to write BoardArbiter:: RotateThisPiece(GrabbedPiece) and I won't need to create a BoardArbiter object. Now, we're thinking without bloated objects!

Naming Importance

▼

    I think I'm good at naming variables and functions. The shape of the characters, the patterns used, and what the words themselves mean. All of these can be combined to create clear and robust naming conventions.

Clear Comments

▼

    I'm glad I was able to work on this project before working on a bigger team. I've developed my own style of how I prefer to write comments, and though I know I'll have to adapt that style to the team's conventions, I think it was very helpful to develop my own first. I'm more worried that I write too many comments, so I think I'll be fine.

    My comments are the pseudo code version of what the code does. The code, then, is merely an implementation of those comments. Thus, you can read the code, without reading any of the actual code.

Pointers

▼

*

    Using pointers effectively was an early hurdle that I needed to get right. I had only learned about them; I hadn't used them much. C++ and Unreal communicate together using them too, but what really helped me was building this project. My backend information needed to be secure if I was going to reliably move pieces around the game without errors. The information needed to stay in one spot. So, with pointers, when the player moves pieces around in the game, they are really just moving pointers to that information. And then, actors in the game are just representations of that information.

Save Games

▼

    Early on, I researched how to implement save games. So, I consolidated all the information that needed to persist in one spot, and the game worked off that consolidated information. A long while passed, and I was ready to actually get save games working. And without too much trouble, I created a custom Save Game Object and successfully saved it and loaded it into a new game. But something was wrong...

    All of my objects were gone! I saved pointers to them, but of course, the objects themselves didn't persist. Although I did spend a lot of time trying to get my code to use references to structs, the solution was to just reinstantiate those objects when reloading the game. I had correctly collected most of the information needed anyway to do this except that I now needed a new way to reference unique pieces. Previously, I was using the pointers themselves as a form of UID. Now, I built my own UID to easily be able to rebuild objects upon loading a saved game. Success!

Material Blueprints and Piece Faces

▼

    There was a key problem I needed to tackle with Material Blueprints, and I was doing it with no prior experience with blueprints or materials. I wanted to take one texture, the full puzzle image, use it on a Material Instance, and to dynamically move the texture into its proper place on each piece. And rotate it accordingly!

    Any variance would mess up the image when you put the pieces together. I wish I was making a game about Squares with the power of 2, but alas, hexagons with a height of sqrt of 12288.

    But I finally got it!

Precise UVs

▼

    The long lesson learned is that their needs to be a tight marriage between models, their UVs, and how the texture is manipulated on the material blueprint, especially when your objects need to be mathematically precise.

Getting Pieces To Recognize Their Connections

▼

    This was one of my largest coding problems on this project. It involved many steps. Each piece needs to store information for each other piece that it connects to. First, the unique ID of the piece it attaches to. Second, the direction of the connection, relative to a base rotation. And finally, what segment of the other piece it needs to be connected to. All of this information is gathered when the puzzles are cut and checked against when a piece is dropped.

Connecting Pieces

▼

    At first, I built the game to move pieces around the board. When I approached the problem of getting pieces to connect together, I knew I had to refactor. So, I introduced Clusters. Now, the player actually moves invisible containers around the board that can hold one or more pieces. Every piece starts with its own cluster. When a connecting piece is found, the piece is added to the initial cluster, and the second one is destroyed.

Rotating Pieces, Rotating Clusters!

▼

    Each spot on the board has a unique address and part of that address is its rotation(Tri). Col, Row, and Tri. A piece can only rotate by 60 degrees, so Tri is represented by an integer(1-6). Now, it's easy enough to rotate a piece.

    It spins around one point, and the Tri changes by 1. If at 7, reset to 1. The rotation of the actor cluster is rotated accordingly, and the pieces are connected to the cluster. But what happens if a large cluster contains pieces in different Col and Row addresses?

    How would I know how to change the addresses of each piece? I first tried to use trigonometry to solve this problem. I got close.

    I was able to reliably get the correct integer change that needed to happen in the Col and Row. Tri still changed only one. However, I couldn't figure out how to know whether I should add this number or subtract this number. Before solving this with trigonometry, I came to realize that I could hard code these numbers with not too much effort. They still followed a pattern, and hard coding this felt more reliable than my decades past AP Trigonometry knowledge.

Full Refactors Are Fun!!

▼

    You commit a stable build to gibhub, make a new branch, and then proceed to break your beloved program. Slowly, you move through the program, changing what you need to. No use trying to compile if you know there will tons of errors. Finally, you've gone through everything you think you need to change. Compile, and you see how you did! And then, you fix minor errors for however long. And usually, you've built your program better than you did last time.

Memory Management / Garbage Collection

▼

    I haven't worked with any low level languages or systems programming, but I have had to understand when I can let memory be garbage collected by UE4 and when and how to protect it(UPROPERTY). So, while I don't have practical experience with memory management, I am starting to scratch the surface of it.

Object-Orientated VS Functional

▼

    Okay. Object-Orientated Programming... After being web taught, OOP was supposed to be the newest, bestest thing. Make everything an object! Turning my attention towards video game programming however, I quickly picked up on the anti-OOP vibe. Being new with UE4 though and programming in general, I struggled just to get things to actually work. Making them work well is another story. Making broad decisions on the different ways to get something to work is acquired over time. My game is not big enough to need to pay attention to every bit, but at least in the scope of a function, I'd be aware of only including actions that are necessary.

 

    In this argument, however, I've recently found that I'm using objects and classes sometimes only as code organization, only as namespaces. My next refactor goal on this project is to remove unneeded objects and replace them with namespace files.

Multiple Levels...Or Not???

▼

    Currently, every puzzle board and every room is a separate level. All pieces, vessels, and backend objects are reloaded and respawned every time you change between rooms and puzzles. I know a lot of this is wasteful. Even though the game runs fine now, I'm weighing the trade offs of moving to just one level. This would mean that I save the comp time of not reloading things for no reason, but it would mean that I'd hold more memory on the heap. Still researching my options.

Blueprint Programming

▼

    I was more intimidated by blueprints than I was with my C++. Over time, I was able to get a better handle on them. I like making them look neat, and I like some of the visual representations of different types of variables. I also like how it's easier to get references from the editor. However, larger operations are better done in C++.

UMG Widget Menus

▼

    Widgets were another Unreal paradigm that took some effort to work with. It didn't help that I came to them without blueprint experience. Following online tutorials weren't that difficult, though. First, I just got one properly working with the game. Next, I put in a pause and stats menu.

Stats Menu

▼

    Making this menu required a good system of communication between the widget and my back-end C++ code. I also had to build a number of hooks in my code so I can gather all of this information. Getting these numbers is one thing, but I also wanted the player to be able to change the look of progress.

    Specifically, that last look, which I've called MatrixBoxes. Initially, I got this working by creating a Widget Image for each of these black squares and making them visible according to a TArray of an Enum that represents each different square. I knew there was probably a better solution, but I was glad I accomplished something “off the beaten path.” Later, I had to add characters to my main Font because it didn't come with some common punctuation.

    Now that my game brought me to learning how to make my own vector-graphic font characters, why don't I make some characters that will act as my boxes? Now, my back-end just returns a string in MatrixBox Language!

Model Rigging And Unreal Integration

▼

    This took me awhile. I've worked with keyframe software before, so doing the simple animation itself wasn't a problem for me. However, I had to learn the basics of rigging 3D models and how to specificallly configure my blender models for exporting into the .fbx file. Then, import them correctly into Unreal. These two animations are minor, but the result of a lot of struggle.

Puzzle Pieces - Randomly Cut And Integrated

▼

    At the beginning of a new puzzle build, the puzzle pieces are randomly cut for each puzzle. This information is stored in simple, custom structs. Objects are made for each of these pieces at the beginning of each session based on this information, and these objects are moved and manipulated throughout the game, which also alters the information in the structs. The on-screen pieces that the player sees are only representations of this information. In the future, I will be able to effect the ratio of smaller pieces to bigger pieces based on difficulty settings.

Game Design/Art

 

Puzzle Images - Curated By Me

▼

    Puzzle images are curated by me, mostly using royalty free images online, but a future goal would be to add images from digital artists. They come in 18 different sizes and currently, there are 234 possible puzzles in the game. Images are picked at random during a world build according to how many pieces or how many puzzles are chosen. Or according to a percentage variable of larger puzzles versus small puzzles.

Vessels

▼

    These vessels represent all the baggage we store our pain in. They are inspired by all the breakable objects in video games that are so fun to clear.

Future Gameplay Verbs

▼

    The other side of the gameplay loop is breaking Vessels, and becoming increasingly better at clearing them. The plan is to create 4 or 5 different types of abilities, which would each have a few levels of magnitude. Each world build would populate vessels with upgrades to these “Vitalities.” Each game would have different combinations of these Vitalities, making for a different type of breaking experience each game. Imagine one game where your stomp is the most powerful. Imagine another where you find lots of items to place that clear full lines of vessels at a time. Imagine another where jumping up clears a lot of vessels at once. AND etc. Here's a mockup of the stats screen that would show different Vitalities gained.
